常用设计模式简述

发表于 2016-08-08 | |

OO中常用的8类设计模式

  1. 策略模式。如果类的一个多个方法会随着不同的业务需求需要频繁的改动而抽象出一个新类用以覆盖原有实现。这时我们需要思考将这些方法一一抽象出一个新的类。而不是不停的使用继承覆盖的形式去改写类的实现。如果OO编程只是继承覆盖继承覆盖的话，那程序的世界还有什么意思，分分钟变蜘蛛网啊，耦合能弄死人啊～～～ 举例而言：我在游戏开发的过程中将npc需要执行的行为抽象为单个的动作对象，主要实现在什么条件下，对谁，在什么时间点，做出什么样的行为。动作的执行判断条件时极为灵活多变的，设想将执行条件一一通过类方法覆盖来实现？不知会做出多少空泛而乏善可陈的类来。我个人最终时采用函数族的形式完成动作执行条件的判断工作。

  2. 观察者模式。映射关系一对多。使用游戏中的业务场景对此进行描述，比如多个npc对一个建筑发动的攻击，则在建筑对象中使用一个容器保存当前对自己发动攻击的npc索引信息。若建筑阵亡则通过上述容器保存的信息，一一通知正在攻击自己的各npc，使之停止攻击。即npc是建筑对象 是否阵亡的观察者。由此我们在建筑阵亡时无需一一通知所有npc。

  1. 发布订阅模式。多对多映射关系。发布者通过平台对象注册自己将会发出的信息类型。订阅者通过平台对象注册自己感兴趣的信息类型。平台对象在收到发布者对象的信息时，依照信息类型通知类型下的订阅者。发布者可取消相遇消息类型的发布，订阅者同样可以取消订阅自己不再感兴趣的消息类型。

  2. 工厂模式。比如在游戏活动中会给玩家送出不同的礼品，礼品对象在实际领取之前通常通过其对应道具类型id来保存的。再如玩家的道具相关数据在存盘后也通常时以道具id，道具类型id来保存。如果我在上述场景中分别依据类型id索取道具类对象，然后将其实例化，代码难移避免的会出现冗余，重叠。此时我们最好统一定义一个工厂类或函数接口，对传入的道具造型即道具信息，返回一个实例化之后道具对象。

  3. 状态模式。对包含状态的对象我们可以将其不同状态下的可执行的方法抽象为一个类，并实例化为含状态对象的一个属性。当对象的状态发生切换时，属性上的对象也发生切换为不同的状态对象，由此影响含有状态的对象的行为。使用状态模式的好处是逻辑清晰，不需要复杂的状态机，低耦合。如在游戏业务的任务场景中可使用，对包含多个不同状态的任务需求可使用状态模式来完成。

  4. 命令模式。代码编写组织以命令为单位抽象为不同的类。每个类含命令的执行方法以及命令的撤销方法。定义统一接口对不同的命令执行不同的命令实例对象。

  5. 迭代器组合模式。对于不同的底层数据结构，可能很多算法的实现代码是一样的，比如迭代便利。这样我门实际无需对不同的容器，一一实现其迭代算法，而可以在更高抽象层次上使用统一的迭代算法对不同的容器结构进行遍历操作。

  6. 装饰者模式。python的装饰器，我个人认为就是装饰者模式的体现。
