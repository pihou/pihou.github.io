年终考核与简历

发表于 2017-01-02 | |

年终总结唰唰的上周就写完了,感觉一般般,和以往书写的套路没太大区别.但是总感觉少了点什么,有点干瘪瘪.和自己的前阵子找工作时的简历的有点类似的味道.想了想又不知道问题出在哪儿.想充实下,却无从下手.写完邮件发送给HR,突然顿悟了,应该是缺少一些量化指标.

比如:

  1. 完成的项目代码量?
    编码量是最容易有效评估的,能一定程度反应工作量以及工作的复杂度.但是我们也需要足够精炼的语言,将项目框架描述清楚

  2. 优化重构后,效率提升数据,并发量?
    对于多数工作内容业务未主的同学而言,这项往往最不敏感.刚入行时,为了完成任务,仓促的只是为了完成任务而已,因为有很多的业务内容需要学习,大家的关注重点放在了业务上.在加之脚本程序语言影藏了太多的细节.我们可以很轻松的,在基本不用关注任何繁琐底层实现细节的情况下,完成任务.
    但是在对业务有了足够的掌握之后,或者说在公司的业务发展到一定量级后,必然会需要,会对程序运行效率有更高的要求,单机能够承载更多的业务量.对程序运行效率的提升,是个人在业务熟练度之上,技术熟练度的体现.
    所以无论什么时候,都应比别人预见更多,准备更多的.

  3. 项目交付后,被交付方工作效率提升数据,评价数据

    或许这项是最难拿到的数据,主观的数据.这项数据通常也是技术人员不太在意的.程序员只用完成指定的活儿就好了.但是实际程序员还很需要全局观,为什么要这么写,为什么要这么做,有没更好的办法.倒不是和产品或策划抬杠什么的,而是我们需要通过寻找这些事情背后的需求,来一定程度的找到自己所做之事的意义和目标.我们需要了解更多,而不是只顾眼前手上的事情.我们需要更多的和他人积极有效的沟通,而不是单单和计算机急速的交流.只有自己的交流足够到位,才能收集到足够多的反馈数据,而这也是我们持续向前,持续改进的动力
