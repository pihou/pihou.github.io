数据库acid

发表于 2018-05-12 | |

数据库事务核心概念 acid

  1. a（atomicity）原子性，即事务内的操作为一个统一整体，要么操作均成功，要么操作均失败；
  2. c（consistency）一致性，即操作前后为统一整体符合实际需求，典型的例子是银行转账；
  3. i（isolation） 隔离，即事务和事务之间形成隔离，操作互不影响；
  4. d（durability）持久性，数据一旦发生修改，将持续有效。

之前对数据库事务的理解一直受限于原子性。好吧，程序世界的名词真的很有限，而概念又太过繁多（再比如容器，tomcat是一种容器，docker是容器技术，list又是一种容器，而且英文统统用的container…）。事务的原子性被自己和操作系统的原子操作相混淆，操作系统的原子操作同时具备了acid的a和i了。

根据自己的理解

  1. 事务的a 更是一个逻辑概念而已，而非技术的实现层面的概念；
  2. 事务的c 和 d 就更是一个业务层面实现的概念了，和使用的具体技术手段实际是隔了好几层的；
  3. 事务的i，则和通常并发编程的常见互斥，同步手段极为接近。

因为这两天有需要又重温了一下数据库的acid，对于自己还没吃透的概念，依旧之志不倦。感谢知乎的这篇问答

至于实现隔离的常规技术手段的核心概念的理解，可以参看Operating Systems: Three Easy Pieces 。 用大白话来介绍下。悲观锁是使用操作系统汇编提供的一条原子指令，这个指令的执行是不可分割，不可被调度中断的（当然还有其它线程，进程同步的技术手段）。测试和设置同时进行，如果被设置的字段和被设置前预计的值不一致，则认为锁已经被他人拿取，反之则自己持有锁。未拿到锁的线程进入阻塞，等待锁的持有者唤醒。乐观锁，在未取得锁的情况下，并不陷入阻塞，而是短暂休眠等待再次获取锁的机会。
