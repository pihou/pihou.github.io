主要用于线程间访问共用内存区域。主要接口及数据类型如下：

pthread_mutex_t类型变量可使用全局内存空间，栈空间，堆空间定义。加锁和解锁操作接口使用了特殊的内部机制实现对锁状态变更的原子操作，而无需使用系统调用。若锁已经处于被锁状态，则使用系统调用（futex() ），将当前线程切换到睡眠状态。互斥锁初始化接口pthread_mutex_init可设置互斥锁类型，以变更互斥锁在不同业务场景下的表现。如：

  1. 对未加锁的互斥锁做解锁操作

linux信号（三）

发表于 2016-08-12 |

一. core dump file

有的信号会导致进程生成一个内存镜像文件，然后退出。内存镜像文件可用于调试，查找宕机的原因。

core dump 文件的生成受限于以下因素：

  1. 进程对其工作目录无写入权限，或其工作目录存在和dump file同名的文件，但是进程对此文件无写权限
  2. 存在和dump file同名的文件，同时进程对其有写权限，但是这个文件存在多个硬连接
  3. 要保存dump file的目录并不存在，即目录要先创建好
  4. 进程本身对生成的core dump文件的大小限制，可使用ulimit -c unlimited 解除限制（RLIMIT_CORE）
  5. 进程本身可生成的文件大小的限制（RLIMIT_FSIZE）
  6. 进程对其对应的可执行文件无读权限，用以防止代码泄漏
  7. 进程的当前工作目录，挂载的文件系统为只读，无剩余空间，i节点用光。用户在此文件系统下的磁盘配合用光了。
  8. set－user－id（set-group-id）程序被非程序所有者执行（但是此时启动的进程存在所有者包含的权限），此时不会产生dump file。防止外人侦探系统敏感信息（如passwords）

常用设计模式简述

发表于 2016-08-08 |

OO中常用的8类设计模式

  1. 策略模式。如果类的一个多个方法会随着不同的业务需求需要频繁的改动而抽象出一个新类用以覆盖原有实现。这时我们需要思考将这些方法一一抽象出一个新的类。而不是不停的使用继承覆盖的形式去改写类的实现。如果OO编程只是继承覆盖继承覆盖的话，那程序的世界还有什么意思，分分钟变蜘蛛网啊，耦合能弄死人啊～～～ 举例而言：我在游戏开发的过程中将npc需要执行的行为抽象为单个的动作对象，主要实现在什么条件下，对谁，在什么时间点，做出什么样的行为。动作的执行判断条件时极为灵活多变的，设想将执行条件一一通过类方法覆盖来实现？不知会做出多少空泛而乏善可陈的类来。我个人最终时采用函数族的形式完成动作执行条件的判断工作。

  2. 观察者模式。映射关系一对多。使用游戏中的业务场景对此进行描述，比如多个npc对一个建筑发动的攻击，则在建筑对象中使用一个容器保存当前对自己发动攻击的npc索引信息。若建筑阵亡则通过上述容器保存的信息，一一通知正在攻击自己的各npc，使之停止攻击。即npc是建筑对象 是否阵亡的观察者。由此我们在建筑阵亡时无需一一通知所有npc。
